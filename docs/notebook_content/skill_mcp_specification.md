# Архитектура и Спецификация Model Context Protocol (MCP)

Model Context Protocol (MCP) — это открытый стандарт, разработанный для обеспечения безопасного и стандартизированного взаимодействия между ИИ-моделями и внешними источниками данных или инструментами. В контексте нашей системы (Функция 5: Управление Скиллами), MCP является ключевым протоколом для реализации динамического обнаружения (Discovery) и ленивой загрузки (Lazy Loading) знаний.

## 1. Проблема, которую решает MCP

До появления MCP интеграция каждого нового инструмента или базы знаний требовала написания индивидуального «клея» (адаптеров). Это приводило к «фрагментации контекста», когда агент не мог эффективно переключаться между разными источниками данных. 
MCP стандартизует этот интерфейс, позволяя модели подключаться к любому «MCP-серверу» и мгновенно получать доступ к его ресурсам и инструментам без переписывания базового кода агента.

## 2. Клиент-Серверная Архитектура

Система MCP строится на разделении ролей:
1.  **MCP Host (Хост)**: Приложение (в нашем случае — ИИ-Агент), которое инициирует соединение. Хост управляет сессиями, правами доступа и промптами.
2.  **MCP Client (Клиент)**: Компонент внутри Хоста, который «говорит» на языке протокола.
3.  **MCP Server (Сервер)**: Легковесный процесс, который предоставляет конкретные возможности. Сервер может находиться локально (stdio) или удаленно (SSE/HTTP).

Эта архитектура позволяет нам изолировать сложные или опасные инструменты внутри отдельных серверных процессов, обеспечивая безопасность основной системы агента.

## 3. Трезубец возможностей: Resources, Prompts и Tools

Протокол MCP определяет три основных способа передачи знаний и способностей модели:

### 3.1. Resources (Ресурсы) 
Ресурсы — это **пассивные данные**, которые сервер предоставляет хосту. Это могут быть файлы, записи в БД или логи. 
- **Метафора**: Это «библиотека», которую модель может читать.
- **Для обучения**: Важно понимать, что ресурсы имеют URI (например, `file://docs/project.md`). Агент может запросить список доступных ресурсов и прочитать их содержимое только тогда, когда это необходимо.

### 3.2. Prompts (Промпты)
Протокол позволяет серверам предоставлять предопределенные шаблоны промптов.
- **Пример**: Сервер для анализа кода может предложить промпт "Объясни эту функцию", который уже содержит оптимальные инструкции для модели.
- **Utility**: Это позволяет «зашивать» лучший опыт взаимодействия прямо в сервер скилла.

### 3.3. Tools (Инструменты)
Это **активные действия**, которые модель может выполнять. Инструментарий MCP полностью совместим со стандартом OpenAI Function Calling.
- Каждый инструмент имеет имя, описание и JSON-схему аргументов.
- Сервер исполняет код инструмента и возвращает результат хосту.

## 4. Протокол связи и Транспорты

Связь между Клиентом и Сервером осуществляется через JSON-RPC 2.0. Это гарантирует надежность и структурированность запросов.

### 4.1. Stdio Transport
Самый популярный и быстрый способ для локальных скиллов. Хост запускает процесс сервера как дочерний процесс и обменивается данными через стандартные потоки ввода-вывода (stdin/stdout).
- **Плюс**: Низкая задержка.
- **Минус**: Сервер должен быть на той же машине.

### 4.2. SSE (Server-Sent Events) Transport
Используется для удаленных инструментов. Позволяет серверу поддерживать открытое соединение с хостом через HTTP.
- **Плюс**: Позволяет создавать облачные библиотеки скиллов (Skill Clouds).

## 5. Жизненный цикл сессии и Безопасность

### 5.1. Initialization (Рукопожатие)
При подключении сервер сообщает свои возможности (Capabilities). Хост проверяет версию протокола и список доступных инструментов. Это момент «Discovery» — агент узнает, чему он может научиться у этого сервера.

### 5.2. Sandboxing (Песочница)
Поскольку инструменты исполняются на стороне сервера, мы можем жестко ограничить их права (например, запретить серверу доступ в интернет или к определенным папкам). В системе LightRAG это критически важно для предотвращения разрушительных действий агента.

## 6. Применение в Управлении Скиллами (Ф5)

Для нашего агента MCP — это способ реализации «Skill Store». 
1.  **Discovery**: Агент сканирует зарегистрированные MCP-серверы.
2.  **Context Optimization**: Вместо того чтобы держать описания всех 100 скиллов в промпте, агент держит там только список серверов. Когда задача требует поиска в базе знаний, он «подключает» нужные ресурсы через MCP Client.
3.  **Standardization**: Мы можем использовать любые готовые MCP-серверы из мирового репозитория (например, серверы для работы с Git, GitHub, Slack, Postgres), и наш агент сразу «поймет», как с ними работать.

## 7. Пример архитектурного пути запроса

1. **User**: "Обнови файл README, добавив описание новой структуры".
2. **Agent**: По URI-схеме ресурсов находит MCP-сервер `filesystem`.
3. **MCP Client**: Запрашивает ресурс `file://README.md`.
4. **Agent**: Получает текст, формирует план изменений.
5. **MCP Client**: Вызывает Tool `write_file` на сервере `filesystem`.
6. **MCP Server**: Выполняет физическую запись и возвращает статус "Success".
