# Методология и Архитектура Плагинов Semantic Kernel

Semantic Kernel (SK) — это SDK с открытым исходным кодом от Microsoft, который позволяет интегрировать LLM с традиционными языками программирования. В нашей системе управления скиллами (Ф5) архитектурные паттерны SK предоставляют эталонную модель того, как превращать разрозненный код в «плагины» — высокоуровневые единицы возможностей, понятные ИИ.

## 1. Понятие Плагина: Больше, чем просто Функция

В терминологии Semantic Kernel, **Плагин** (Plugin) — это контейнер для набора функций, сгруппированных по смыслу или домену. 
Если Function Calling занимается отдельными вызовами, то Плагин — это полноценный программный интерфейс для ИИ.

Ключевой принцип: **Инкапсуляция**. Плагин скрывает сложность реализации (будь то SQL-запрос, вызов внешнего API или расчет по формуле) за семантическим описанием.

## 2. Типы Функций внутри Плагина

### 2.1. Native Functions (Нативные функции)
Это функции, написанные на классических языках (C#, Python, JS). Они выполняют детерминированные задачи: расчеты, манипуляции с файлами, сетевые запросы.
- **Архитектурная роль**: "Руки" агента.
- **Требование**: Каждая нативная функция должна иметь декоратор (аттрибут) с описанием. В SK это выглядит как `[KernelFunction]` и `[Description("...") ]`. Это описание напрямую попадает в промпт модели.

### 2.2. Semantic Functions (Семантические функции)
Это функции, логика которых реализована на естественном языке (промпты). Они выполняют вероятностные задачи: суммаризация, перевод, анализ тональности.
- **Архитектурная роль**: "Мысли" агента внутри инструмента.
- **Преимущество**: Мы можем упаковать сложный промпт (например, "Проверь этот текст на соответствие стилю бренда") как простую функцию, которую агент может вызвать так же, как и код.

## 3. Переменные и Контекст (Kernel Arguments)

Одной из самых мощных концепций SK является управление аргументами через `KernelArguments`. 
- Инструменты не просто получают статичные данные. Они могут черпать информацию из общего «контекстного окна» ядра (Kernel).
- Переменные могут передаваться по цепочке: результат одной функции автоматически становится входом для другой. Это позволяет создавать **сложные композитные скиллы**.

## 4. Паттерн "Planner": Автоматическая композиция

Semantic Kernel ввел понятие **Планировщика** (Planner). Вместо того чтобы мы вручную прописывали последовательность действий, агент-менеджер может:
1.  Получить цель от пользователя.
2.  Просканировать список доступных плагинов (их описания).
3.  Самостоятельно составить «план выполнения», решив, в какой последовательности вызывать функции.

Для Ф5 это означает, что если в базе есть атомарные скиллы "Читать файл", "Перевести" и "Записать файл", агент сможет сам синтезировать из них новый навык "Локализация документа", просто понимая семантику каждого плагина.

## 5. Стандарты описания для максимального понимания LLM

Для того чтобы обучение в LightRAG было качественным, каждый плагин должен следовать «золотому стандарту» метаданных SK:
- **Название плагина**: Должно быть существительным, отображающим домен (например, `FileIOPLugin`, `TimePlugin`).
- **Название функции**: Глагол, описывающий действие (например, `ReadText`, `GetCurrentTime`).
- **Описание параметров**: Каждый входной параметр должен иметь описание и тип. Если параметр имеет значения по умолчанию — это должно быть указано.
- **Input/Output типизация**: Использование сложных типов (классов/объектов) должно сопровождаться схемой, чтобы модель знала внутреннюю структуру данных.

## 6. Изоляция и Dependency Injection (DI)

Современная архитектура SK-плагинов требует использования внедрения зависимостей. 
- Плагин не должен знать о деталях реализации системы. 
- Если плагину нужно отправить письмо, он должен запрашивать интерфейс `IEmailService`.
- Это позволяет нам менять «под капотом» реализацию скилла (например, сменить провайдера отправки почты), не меняя документацию и знания агента. Агент продолжает вызывать ту же функцию, но она работает эффективнее.

## 7. Применение в LightRAG: Построение Иерархии Скиллов

Используя опыт Semantic Kernel, мы строим в нашей базе иерархию:
1.  **Level 0: Атомарные функции** (чтение, запись, поиск).
2.  **Level 1: Специализированные плагины** (MCP Servers, Python Scripts).
3.  **Level 2: Оркестрация (Planners)** — мета-знание о том, как комбинировать плагины для достижения бизнес-целей.

Благодаря такой структуре, наш агент перестает быть «скриптом» и становится динамической системой, способной к саморазвитию через добавление новых плагинов в библиотеку.
