# Глубокое руководство: OpenAI Function Calling и Инженерия Инструментов

Этот документ представляет собой фундаментальный разбор механизмов вызова функций (Function Calling) и инструментов (Tools) от OpenAI. Эти знания критически важны для реализации Функции 5 (Управление Скиллами) в рамках автономного ИИ-агента, так как они определяют границу между «рассуждением» модели и её «действием» в реальном мире.

## 1. Концепция: Код как Действие (Action Space)

Традиционные LLM ограничены текстовым выводом. Function Calling расширяет их возможности, позволяя модели не просто «говорить», а формировать структурированные запросы к внешним системам. В этой архитектуре модель берёт на себя роль **планировщика**, а код/API — роль **исполнителя**.

Ключевой сдвиг парадигмы: модель не должна сама исполнять код. Она должна генерировать аргументы, которые соответствуют предопределенной схеме (JSON Schema), гарантируя, что внешний исполнитель поймёт команду.

## 2. Анатомия Tool Definition (args_schema)

Каждый инструмент (скилл) должен быть описан с предельной точностью. Ошибка в описании — это «галлюцинация действия».

### 2.1. Описание функции (Description)
Описание функции — это промпт для модели, объясняющий, *когда* и *зачем* использовать этот инструмент. 
- **Плохо**: `search_web(query)` — "Ищет в интернете".
- **Хорошо**: `search_web(query, domain_filter)` — "Использует поисковый движок для получения актуальной информации из сети. Используйте этот инструмент, когда пользователь спрашивает о текущих событиях или данных, которых нет в ваших весах обучения. Параметр domain_filter позволяет сузить поиск до конкретных сайтов (например, 'github.com' для поиска кода)".

### 2.2. JSON Schema и Типизация
OpenAI использует подмножество JSON Schema для описания аргументов. Для качественного обучения базы LightRAG мы выделяем следующие обязательные поля:
- `type`: 'object', 'string', 'number', 'boolean', 'array', 'enum'.
- `properties`: Детальное описание каждого поля.
- `required`: Список обязательных полей. Без них модель может пропустить важный контекст.
- `items` (для массивов): Определение типа содержимого списка.

### 2.3. "Тест для Интернa"
При составлении `args_schema` используйте принцип: "Поймет ли стажер-программист, что сюда передать, не видя исходного кода?". Описания полей должны включать форматы (например, "ISO 8601 для дат") и ограничения.

## 3. Механика взаимодействия: Сообщение 'Tool Call'

Взаимодействие происходит через специфический тип сообщений в чате:
1. **User**: Запрос ("Какая погода в Лондоне?").
2. **Assistant**: Генерирует `tool_calls`. Это не текст, а JSON-объект с именем функции и аргументами.
3. **System/Developer**: Исполняет функцию локально (через ваш код) и возвращает результат в виде сообщения с ролью `tool`.
4. **Assistant**: Получает результат (например, "20 градусов") и генерирует финальный ответ пользователю ("В Лондоне сейчас 20 градусов").

**Важно**: Модель не имеет доступа к интернету или API напрямую. Она живет в «текстовой песочнице», и именно вы являетесь её руками в реальном мире.

## 4. Продвинутые паттерны и Ограничения

### 4.1. Параллельный вызов функций (Parallel Function Calling)
Современные модели (gpt-4o, gpt-4-turbo) способны вызывать несколько функций одновременно. Например, если пользователь просит "Сравни погоду в Киеве и Берлине", модель сгенерирует два `tool_calls` в одном ответе. Скилл-менеджер должен уметь обрабатывать такие списки параллельно для ускорения работы.

### 4.2. Использование 'Strict Mode'
Включение `strict: true` в определении функции заставляет модель строго следовать JSON-схеме. Это снижает риск галлюцинаций в типах данных (например, когда модель пытается передать строку в поле, помеченное как `number`). Рекомендуется всегда использовать этот режим для критических систем управления файлами или базами данных.

### 4.3. Ограничения контекста
Каждое определение функции занимает токены в контекстном окне. 
- **Проблема**: Если у вас 100 сложных скиллов, контекстное окно переполнится, и модель станет менее точной.
- **Решение (JIT-загрузка)**: Скилл-менеджер должен выполнять динамический поиск (Discovery) и подгружать в промпт только те 5-10 инструментов, которые наиболее релевантны текущему запросу пользователя.

## 5. Обработка ошибок и Рефлексия

Если выполнение функции завершилось ошибкой (например, API вернуло 404), вы должны передать это сообщение об ошибке обратно модели с ролью `tool`. 
Хорошо обученная модель (использующая паттерн Reflexion) увидит ошибку, проанализирует её ("Ага, я передал неверный ID") и совершит повторную попытку с исправленными параметрами. Это самоисцеляющийся цикл управления скиллами.

## 6. Рекомендации по Инкапсуляции

Каждый скилл в системе LightRAG должен быть оформлен как независимый модуль, содержащий:
1. `manifest.json` — Описание для LLM (Schema).
2. `logic.py` — Исходный код выполнения.
3. `documentation.md` — Подробное руководство для агента (которое сейчас и индексируется).

Такой подход обеспечивает модульность и позволяет агенту легко «учить» новые навыки, просто прочитав их документацию и добавив их в свою библиотеку.
