# Архитектура Skill Library: Опыт агента Voyager (Code-as-Action)

Проект Voyager (разработанный исследователями NVIDIA и университетов) стал революцией в области автономных агентов. В отличие от традиционных моделей, которые полагаются на текстовые описания действий, Voyager использует **код как пространство действий** (Code-as-Action). Этот документ описывает архитектуру «Библиотеки Навыков» (Skill Library), которая позволяет агенту накапливать процедурный опыт и становиться умнее с каждой решенной задачей.

## 1. Проблема "Кусочного Опыта"

Типичные ИИ-агенты решают задачи "в моменте": они генерируют план, выполняют его и забывают детали реализации. Для решения сложной задачи в следующий раз им приходится начинать планирование с нуля. 
Voyager решает это через **Skill Library** — персистентное хранилище успешно выполненных программ.

## 2. Три столпа архитектуры Voyager

Для понимания управления скиллами мы выделяем три компонента системы:
1.  **Automatic Curriculum (Автоматическая учебная программа)**: Ставит цели повышения сложности.
2.  **Iterative Prompting Mechanism**: Реализует цикл написания и дебаггинга кода скилла.
3.  **Skill Library**: Накопитель программного кода, индексируемый семантически.

## 3. Как работает Skill Library (Технический разрез)

### 3.1. Индексация: Векторный поиск по смыслам
Библиотека представляет собой векторную базу данных (например, Pinecone, Milvus или наш Qdrant/Neo4j). 
- **Key (Ключ)**: Эмбеддинг (векторное представление) текстового описания того, что делает этот навык (например, "Как скрафтить деревянную кирку").
- **Value (Значение)**: Исполняемый программный код (JavaScript/Python), который реализует это действие.

### 3.2. Процесс Retrieval (Извлечение)
Когда агент получает новую цель, он делает запрос к библиотеке: "Найди мне 5 самых полезных навыков для текущей ситуации". 
Система возвращает не просто текст, а **подходящие фрагменты кода**, которые агент может вызвать как функции внутри своей новой программы. Это позволяет избежать повторного изобретения колеса.

## 4. Жизненный цикл нового скилла: От ошибки к библиотеке

Самый важный урок Voyager для нашей системы Ф5 — это процесс превращения одноразового действия в постоянный навык:

1.  **Drafting**: Агент пишет черновик кода для новой задачи.
2.  **Execution & Feedback**: Код запускается в среде. Если возникают ошибки (traceback) или среда сообщает о неудаче, агент получает эти данные обратно.
3.  **Self-Verification (Критик)**: Модель GPT-4 анализирует логи и визуальное состояние (или статус системы) и выносит вердикт: "Задача выполнена успешно" или "Нет, ты забыл взять предмет".
4.  **Commitment (Фиксация)**: Только если задача выполнена на 100%, агент генерирует чистое описание этого действия и сохраняет пару {Описание : Код} в Skill Library.

## 5. Композиция навыков: Строительство Собора из Кирпичей

Архитектура Voyager поощряет иерархичность. 
- Навык `mine_wood` становится подфункцией для навыка `craft_planks`.
- Навык `craft_planks` используется в `craft_table`.
В библиотеке накапливается дерево зависимостей. Это превращает агента из «генератора текста» в «программного инженера», который оперирует абстракциями высокого уровня.

## 6. Почему это критично для LightRAG и Ф5?

Обучая нашу базу на опыте Voyager, мы внедряем в менталитет ИИ-агента следующие правила:
- **Не пиши код с нуля**, если в библиотеке есть похожий.
- **Инструмент считается инструментом только тогда, когда он верифицирован**. Нельзя добавлять в Ф5 «битые» или непроверенные пути.
- **Каждый успех должен быть задокументирован**. Если решение сработало один раз, оно должно быть сохранено для будущего использования через механизм JIT-индексации.

## 7. Практическая реализация "Skill Library" в нашем проекте

Для реализации этого в LightRAG мы используем:
- **`docs/experience/`** как физическое хранилище "проверенных" подходов (Value).
- **GraphRAG (Neo4j)** для создания связей между навыками и контекстами их применения (Key/Metadata).
- **JIT Agentic Ingestion** как механизм верификации (тот самый "Критик" из Voyager).
