# Практическая Имплементация Циклов Самокоррекции в LangGraph

LangGraph — это расширение LangChain, предназначенное для создания циклических графов (State Machines), которые идеально подходят для реализации паттерна Reflexion. В то время как обычные цепочки (Chains) прямолинейны, LangGraph позволяет агенту возвращаться назад для исправления ошибок. Для нашей функции управления скиллами (Ф5) это основной способ реализации «настойчивости» и качества выполнения.

## 1. Концепция: Агент как Граф Состояний (State Machine)

В LangGraph агент — это не просто последовательность вызовов, а набор **Узлов** (Nodes) и **Ребер** (Edges), которые передают между собой объект `State`.
`State` (Состояние) содержит в себе:
- Историю сообщений (Messages).
- Результаты работы инструментов (Tool Outputs).
- Текст текущей рефлексии (Critique).
- Счётчик итераций (Max Loops).

## 2. Ключевые узлы архитектуры Reflexion

Для реализации самокоррекции мы выделяем три обязательных узла:

### 2.1. Узел 'Responder' (Исполнитель)
Этот узел принимает текущее состояние и генерирует «черновик» решения или вызывает необходимые инструменты (через OpenAI Function Calling).
- Если это первая итерация, он работает по запросу пользователя.
- Если это повторная итерация, он получает в контексте критику от узла `Revisor` и обязан внести правки.

### 2.2. Узел 'Revisor' (Критик/Ревизор)
Самый важный узел для Ф5. Он получает результат от `Responder` и выполняет проверку:
- "Соответствует ли ответ техническому заданию?"
- "Нет ли в коде синтаксических ошибок?"
- "Верно ли заполнены все обязательные поля инструмента?"
Его задача — не исправлять код самому, а написать жесткую, конструктивную критику для `Responder`.

### 2.3. Узел 'Tools' (Исполнение инструментов)
Если `Responder` решил вызвать MCP-инструмент, управление передается в узел `Tools`. Результаты выполнения возвращаются в состояние и становятся доступны Ревизору для оценки.

## 3. Ребра и Логика Маршрутизации (Conditional Mapping)

Ребра в LangGraph определяют путь данных. В цикле рефлексии мы используем **условные ребра**:
1. После работы Ревизора граф спрашивает: "Достаточно ли хорош результат?".
2. Если **ДА** (или достигнуто заложенное качество) -> переход к завершению (END).
3. Если **НЕТ** -> возврат в узел `Responder` для следующей итерации.

## 4. Управление "Памятью Ошибок"

Чтобы цикл рефлексии не стал «бесконечным бегом по кругу» (когда агент повторяет одну и ту же ошибку), мы применяем стратегию накопления:
- В объекте `State` мы храним не все сообщения, а **суммированную историю рефлексий**. 
- При каждой новой попытке Исполнитель видит: "Попытка 1: Ошибка A. Попытка 2: Ошибка B". 
- Это заставляет модель менять стратегию (например, сменить инструмент или значительно переписать параметры).

## 5. Предохранители: Max Iterations и Timeouts

При управлении скиллами агент может попасть в ловушку (например, инструмент постоянно возвращает 500 ошибку). 
- Мы внедряем в граф переменную `loop_count`. 
- Если количество итераций превышает предел (например, 3 или 5), граф принудительно направляет поток в узел "Human-in-the-loop" или выдает отчет о невозможности выполнения с детальной рефлексией причин.

## 6. Преимущества для Ф5 в системе LightRAG

Обучая агента на этом документе, мы даем ему ментальную модель «Рабочего Процесса»:
- **Сначала подумай, потом сделай**.
- **Проверь себя (или дай проверить другому узлу)**.
- **Ошибаться нормально, если ты можешь объяснить ошибку и исправить её**.

Этот документ является техническим чертежом того, как Скилл-Менеджер должен координировать вызовы инструментов, обеспечивая 100% надежность даже при работе с нестабильными внешними системами.
