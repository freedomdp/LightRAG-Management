# 🕸️ Сфера Ф8: Узлы, Графы и Многоагентная Специализация

## 1. Нодовое мышление (n8n Standard)
Вместо восприятия проекта как бесконечного текстового потока, Агент должен видеть его как набор связанных узлов (Nodes). Каждый узел — это изолированная функция.

### Принципы нодовой архитектуры:
- **Isolation**: Изменения внутри одного узла не должны ломать логику других, если интерфейсы (Input/Output) сохранены.
- **Visual Mapping**: Возможность представить поток данных в виде схемы, где каждый «кубик» — это конкретный скилл или микрозадача.
- **Error Handling**: Каждый узел имеет ветку `on_error`, что позволяет системе самовосстанавливаться без краха всего процесса.

## 2. Паттерн «Планировщик - Исполнитель» (LangGraph)
Чтобы не «сходить с ума» от объема данных, Агент делится на роли.

### Роль A: Planner (Стратег)
- Видит «Карту репозитория» (Repo Map).
- Составляет JSON-чеклист подзадач.
- Не пишет ни одной строки кода.
- **KPI**: Точность детализации и предсказание зависимостей.

### Роль B: Workers (Скиллы)
- Видят только 1-2 файла, необходимых для задачи.
- Имеют узкий набор инструментов (например, только `grep` и `edit`).
- Выполняют задачу по шагам чек-листа.
- **KPI**: Прохождение тестов и соответствие Linter-правилам.

## 3. Механизм делегирования (Hand-offs)
Передача управления между ролями происходит через «архитектурный контракт» (например, запись плана в `implementation_plan.md`). Исполнитель не может менять план Стратега без инициации нового цикла планирования.

## 4. Иерархия управления контекстом
1. **Global Graph**: Высокоуровневая структура проекта. (Знает Planner).
2. **Local Scope**: Текущий редактируемый модуль. (Знает Worker).
3. **Atomic Edit**: Конкретный блок кода. (Фокус внимания в моменте).

## 5. Защита от «Универсализма»
Агент-универсал — это путь к галлюцинациям. Субъектность архитектора заключается в умении вовремя ограничить свой контекст (Context Pruning) и перейти в роль узкоспециализированного воркера.
