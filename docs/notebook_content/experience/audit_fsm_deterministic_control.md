# ⚙️ Сфера Ф8: Детерминированный контроль через State Machines

## 1. Проблема «Свободной воли» Агента
Чистая LLM-оркестрация склонна к «прыжкам» через этапы (Step Skipping) и потере промежуточных валидаций. Для обеспечения субъектности Агент должен воспринимать процесс как конечное множество состояний (FSM - Finite State Machine).

## 2. Архитектура Графа состояний (DAG)
Любая сложная задача (например, «напиши фичу») должна быть разложена на направленный ациклический граф (DAG), где переход к следующему узлу невозможен без выполнения условий текущего.

### Типовые состояния (States):
- **IDLE**: Ожидание задачи.
- **PLANNING**: Написание `implementation_plan`. Переход только после `request_feedback = true`.
- **RESEARCH**: Анализ кодовой базы. Переход после фиксации всех зависимостей.
- **EXECUTION**: Написание кода. Переход только после завершения всех правок в `task.md`.
- **VERIFICATION**: Тестирование. Переход к DONE только при `exit code 0`.

## 3. Протокол валидации переходов (Transition Rules)
Агент не «придумывает» следующий шаг, а сверяется с узлами чек-листа:
1. **Input Validation**: Данные получены?
2. **Action**: Работа выполнена?
3. **Output Gate**: Результат проверен (Linting/Tests)?
4. **Transition**: Если Да -> Переход. Если Нет -> Цикл исправления (Retry).

## 4. Отсечение импровизации (Zero Fantasy Rule)
Если текущее состояние — `CODE_EDITING`, Агент физически заблокирован от того, чтобы начать «рефакторить другие файлы» или «обновлять README», если это не входит в текущий узел графа. Субъектность заключается в осознанном ограничении своих возможностей ради предсказуемости результата.

## 5. Преимущества для Архитектора
Бизнес-логика выносится из промптов в структуру графа (например, в n8n или LangGraph). Это делает разработку AI-систем воспроизводимой (Reproducible) и управляемой.
