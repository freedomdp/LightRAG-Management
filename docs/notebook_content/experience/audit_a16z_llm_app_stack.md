# 🏛️ Сфера Ф8: Архитектурные слои LLM-приложений (a16z Standard)

## 1. Концепция многослойности
Согласно эталонной архитектуре a16z (Emerging Architecture for LLM Apps), современное интеллектуальное приложение не является монолитом промптов. Оно состоит из трех изолированных слоев, взаимодействие между которыми определяет стабильность системы.

### А. Слой Логики и Оркестрации (Logic Layer)
Это «мозг» системы, где происходит декомпозиция задач, планирование и маршрутизация.
- **Инструменты**: LangChain, LangGraph, Haystack.
- **Роль Агента**: Здесь Агент выступает как *Planner*. Он не пишет код в вакууме, а оркеструрует вызовы инструментов.
- **Mandatory Protocol**: Любое решение Агента на этом слое должно быть верифицировано через 'Logic Grader'.

### Б. Слой Данных и Памяти (Data/Memory Layer)
Слой, отвечающий за контекст и долгосрочное хранение знаний.
- **Компоненты**:
    - *Vector DB (Qdrant)*: Семантическое облако для быстрого поиска.
    - *Graph DB (Neo4j)*: Структурные связи между сущностями.
- **Практика**: Агент обязан осознавать разницу между «поиском по сходству» (Vector) и «поиском по связям» (Graph). Галлюцинации часто возникают при попытке извлечь структурные данные через векторный поиск.

### В. Слой Инструментов и Действий (Tool/Control Layer)
Слой взаимодействия с внешним миром (API, Файловая система, Песочницы).
- **Принцип изоляции**: Успешный агент не пытается «решить задачу генерацией», если есть инструмент. Если нужно прочитать файл — вызывается `view_file`, а не галлюцинируется его содержимое.

## 2. Разделение обязанностей (Principle of Least Authority)
Агент верхнего уровня (Архитектор) имеет доступ к планированию, но не должен выполнять «грязную» работу. Он делегирует её специализированным воркерам (Skills). Это минимизирует когнитивную нагрузку и предотвращает "Fog of War" (затуманивание контекста).

## 3. Практические метрики (KPI Архитектора)
Для оценки качества архитектуры Агент должен отслеживать:
1. **Latency (Задержка)** на каждом слое.
2. **Retrieval Precision**: Насколько найденные чанки релевантны задаче.
3. **Execution Robustness**: Процент успешных вызовов внешних инструментов.

## 4. Иерархия принятия решений
1. **Goal Analysis**: Анализ цели и выявление ограничений (Constraints).
2. **Layer Assignment**: Определение, какой слой (Logic/Memory/Tool) должен быть задействован.
3. **Drafting**: Создание плана (implementation_plan).
4. **Execution**: Запуск цепочки через State Machine (см. след. мануал).
