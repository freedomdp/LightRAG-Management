# Спецификация: Архитектура Многоуровневой Памяти (Letta/MemGPT Pattern)

## Концепция: Агент как Операционная Система
Разработчики MemGPT (Letta) предлагают рассматривать память агента не как плоский список сообщений, а как иерархическую систему хранения:

1. **Short-term Memory (STM / L1 Cache)**:
   - Текущее «окно контекста» (последние N сообщений).
   - *Характеристики*: Быстрый доступ, высокая стоимость токенов.

2. **Working Context (Memory Buffer)**:
   - Специально отобранные факты о проекте, которые агент «держит в уме» постоянно.
   - *Для LightRAG*: Это файл `STATUS.md` и переменные окружения.

3. **Long-term Memory (LTM / Disk)**:
   - База знаний Neo4j/Qdrant. 
   - *Доступ*: Только через семантический поиск (RAG).

## Стратегия управления контекстом (Eviction Policies)
Главный враг агента в IDE — переполнение контекста лишними деталями.
- **Selective Retrieval**: Агент не должен «выгружать» всю историю чата. Вместо этого он должен запрашивать из LTM только те «отпечатки опыта», которые относятся к текущему файлу или функции.
- **Archival Trigger**: Когда STM переполняется, агент обязан провести «дистилляцию» знаний и сохранить ключевые выводы в LTM, очистив окно контекста для новых задач.

## Предотвращение «Галлюцинаций Переполнения»
При достижении 80% лимита контекстного окна (например, в Claude), агент должен:
1. Выполнить `log_new_experience` для текущей задачи.
2. Сбросить лишние детали.
3. Продолжить работу, опираясь на свежесозданную «точку сохранения» в опыте.

## Реализация в Antigravity
- **Recursive Reasoning**: Каждое действие в `<logic_tree>` должно учитывать иерархию памяти. 
- **Context Awareness**: Если задача требует изменения 50 файлов, агент не читает их все. Он использует LTM (Neo4j) для понимания графа зависимостей и «подкачивает» в STM только нужные фрагменты.

## Золотое правило Letta
*"Агент не должен помнить всё. Агент должен знать, где найти всё."*
