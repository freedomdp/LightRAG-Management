# Спецификация: Категоризация Памяти: Semantic vs User Preference (OpenSearch Pattern)

## Введение
OpenSearch предлагает строгую типизацию агентной памяти для повышения точности RAG-извлечения. Размытие границ между "как работает система" и "как хочет пользователь" ведет к архитектурным ошибкам.

## Типы Памяти (Strict Typing)

1. **SEMANTIC MEMORY (Инженерный опыт)**:
   - Знания о коде, архитектуре, библиотеках и баг-фиксах.
   - *Цель*: Техническая корректность.
   - *Пример*: Правильное использование `nohup` или `os.replace`.

2. **USER_PREFERENCE MEMORY (Вайб-память)**:
   - Знания о стиле, предпочтениях и "вайбе" разработчика.
   - *Цель*: Удовлетворенность пользователя (User Bliss).
   - *Пример*: Использование конкретного стиля коммитов или Markdown-разметки.

3. **PROCEDURAL MEMORY (Методология)**:
   - Знания о том, в каком порядке выполнять задачи (чек-листы).
   - *Цель*: Соблюдение стандартов проекта.

## Механика Извлечения (Context Selection)
Агент должен «знать свою роль». 
- Если он **Кодит** ➔ Приоритет Semantic Memory.
- Если он **Общается** ➔ Приоритет User Preference.
- Если он **Планирует** ➔ Приоритет Procedural Memory.

## Разделение в LightRAG
Мы используем теги и структуру атрибутов в Neo4j для обеспечения этого разделения. 
- *Инструкция*: При поиске опыта всегда учитывайте тип задачи (Plan vs Exec vs Talk).

## Правила для Antigravity
- Не смешивайте технический фикс с изменением стиля общения в одной записи `log_new_experience`. 
- Делайте записи гранулярными. Лучше две маленькие и точные записи, чем одна большая и расплывчатая.
