# 📑 Сфера Ф8: Инженерный стандарт трассировки (Langfuse)

## 1. Иерархия логов
В отличие от обычного текстового логирования, Langfuse вводит строгую иерархию объектов, позволяющую анализировать выполнение сложных агентных цепочек.

- **Trace**: Корень операции. Должен иметь `name` (напр. `feature_implementation`) и `user_id`.
- **Observations**: Все, что происходит внутри трейса. Делятся на:
    - **Spans**: Логические единицы работы (напр., `sql_generation`, `code_linting`).
    - **Generations**: Сами вызовы LLM с полным промптом и ответом.
    - **Events**: Атомарные события (напр., «Файл сохранен», «Ошибка получена»).

## 2. Скоринг и Валидация
Langfuse позволяет прикреплять «Оценки» (Scores) к генерациям.
- **Auto-eval**: Система автоматически проверяет, содержит ли ответ JSON или запрещенные слова.
- **Human-in-the-Loop**: Разработчик может поставить 'thumbs up/down', что попадает в базу знаний Агента как «Успешный/Неуспешный опыт».

## 3. Мониторинг задержек (Latency Analysis)
Агент обязан логировать время выполнения каждого шага. Если шаг `research` занимает более 30 секунд, Агент должен проанализировать причину (ожидание API или слишком большой объем данных).

## 4. Контроль версий промптов
Все промпты, используемые Агентом, должны иметь `prompt_tag` (напр. `v1.2-stable`). Это позволяет сопоставлять результаты и понимать, какая версия инструкций ведет к галлюцинациям.

## 5. Практическое применение
При выполнении задачи Агент имитирует структуру Langfuse в своем `walkthrough.md`, визуализируя вложенность задач для архитектора.
