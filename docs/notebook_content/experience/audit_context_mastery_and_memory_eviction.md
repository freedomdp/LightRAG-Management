# 🧠 Сфера Ф10: Умное управление контекстом и памятью

## 1. Проблема «Затуманивания контекста» (Context Bloat)
При загрузке всего репозитория в промпт Агент тратит когнитивный ресурс на шумные данные, что ведет к галлюцинациям. Решение — использование **Repository Maps** (Aider Standard).

## 2. Repo Maps и AST-деревья
Вместо чтения файлов целиком, Агент должен оперировать «картой сигнатур».
- **Repo Map**: Список всех файлов с указанием публичных классов и функций (без тел функций).
- **AST (Abstract Syntax Tree)**: Позволяет Агенту понять структуру зависимостей (кто кого вызывает), не открывая сами файлы.
- **Protocol**: При входе в проект Агент сначала генерирует/обновляет карту, находит релевантные узлы и только потом запрашивает полный контент 2-3 файлов через `view_file`.

## 3. Стратегии вытеснения памяти (Memory Eviction / PostHog)
Когда история чата забивает окно контекста (Context Window), Агент применяет активную очистку по стандартам PostHog.
- **Sliding Window**: Сохранение только последних N сообщений.
- **Semantic Pruning**: Удаление промежуточных рассуждений и ошибок, сохранение только «финальных фактов» и принятых решений.
- **Recursive Summarization**: Старые части диалога сжимаются в один абзац-сапмари.

## 4. Группировка по версиям контекста (Context Versioning)
Агент должен помечать каждый свой ответ «версией контекста» (напр., какие файлы были открыты в этот момент). Это позволяет откатиться к состоянию, когда логика была еще «чистой» (до возникновения дрейфа).

## 5. RAG vs. Active Memory
- **Knowledge Base (LightRAG)**: Долгосрочная память (библиотеки, стандарты). Доступ через поиск.
- **Active Context**: Краткосрочная память (текущие значения переменных, открытые файлы). Доступ мгновенный.
- **Optimization**: Агент обязан переносить важные осознания из кратковременной памяти в долгосрочную через `log_new_experience`.
