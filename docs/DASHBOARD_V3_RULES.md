# Спецификация и Правила Формирования Дашборда (STATUS.md) V3.0

Этот документ описывает архитектуру, структуру и правила обновления файла `STATUS.md`, который служит главной приборной панелью для мониторинга системы LightRAG. Скрипт мониторинга должен запускаться каждые 60 секунд.

## 1. Структура `STATUS.md`

Дашборд должен атомарно (целиком) перезаписываться скриптом каждые 60 секунд, строго сохраняя следующую структуру:

### 1.1. Шапка и время обновления
В самом верху документа всегда должна быть строка с точным временем последнего обновления.
**Формат:**
`Обновлено: 2026-04-11 00:20:43 (Авто-обновление: 60 сек)`

### 1.2. Раздел ⚡ Статус Индексации
Этот раздел должен отражать текущее состояние системы обработки текстов. Обязательные элементы:
- **Общий статус системы**: Нужно показывать, чем сейчас занята система (например, Индексация, Завершение, Покой, 🛑 ЗАВИСАНИЕ (Требуется вмешательство)).
- **Прогресс документов**: Количество полностью обработанных файлов к общему количеству (например, `[241/442]`). Дополнительно можно выводить процент.
- **Прогресс частей (Чанков)**: Количество обработанных частей документа к общему количеству частей в *текущем активном* документе (например, `[3/12]`).
- **Активность LLM (GPU/Ollama)**: Состояние языковой модели (работает, простаивает 💤, перегружена).

### 1.3. Раздел 🖥️ Системные ресурсы (Docker & Mac)
Данные о потреблении ресурсов агрегируются из `docker stats` и системных утилит (например, `ps` для Ollama).
Обязательная таблица со столбцами:
`| Компонент / Контейнер | Нагрузка (CPU %) | RAM / Mem | Статус |`

Обязательные строки:
1. **Ollama (Metal GPU)**: Вычисляется по процессу macOS.
2. **LightRAG API** (`lightrag_api`): Данные из Docker.
3. **Neo4j DB** (`lightrag_neo4j`): Данные из Docker.
4. **Qdrant DB** (`lightrag_qdrant`): Данные из Docker.

**Формат статуса:** `✅ Active`, `✅ Online`, `🛑 Offline`, `⚠️ High Load`.

### 1.4. Раздел 📜 Недавняя активность (Самые новые - сверху)
Журнал событий, предоставляющий прозрачную картину работы системы.
**Правила ведения журнала:**
- **Сортировка**: Новые события всегда добавляются ВЕРХ (индекс 0), старые смещаются вниз.
- **Вместимость**: Не менее 100 последних событий. Старые (за пределами 100-150) аккуратно удаляются/ротируются.
- **Формат времени**: В начале каждой строки обязательно идет время `[12:35:05]`.
- **Типы событий (Обязательно для захвата)**:
  - Обработка документа завершена: `[12:35:05] 📄 Документ обработан: file_name.md`.
  - Обработка части документа (чанка): `[12:35:00] 🧩 Обработан чанк [3/12] в документе file_name.md`.
  - Запуски и паузы скриптов: `[12:30:00] ▶️ Запуск скрипта индексации`.
  - Ошибки и перезапуски.

## 2. Логика Источников Данных (Откуда брать параметры)

1. **Документы и Чанки**: 
   Параметры должны парситься из логов LightRAG API (`docker logs lightrag_api`) и/или локального файла статуса `temp/ingest_status.json`.
   Для чанков парсить лог-записи вида `Chunk 3 of 12`.
   **Таймаут (Freeze-контроль)**: Скрипт должен сравнивать временную метку последнего изменения метрик (документа или чанка) с текущим временем. Если простой превышает 5 минут (300 сек), фиксируется зависание работы LLM/API.
2. **Системные Ресурсы**:
   Опрашивать `docker stats --no-stream --format "{{.Name}}:{{.CPUPerc}}:{{.MemUsage}}"` для контейнеров.
   Опрашивать `ps -A -o %cpu,command | grep ollama | grep -v grep` для Ollama.
3. **Журнал Активности**: 
   Скрипт должен хранить состояние журнала (state) локально, например, в JSON-файле `temp/dashboard_log.json`, чтобы при перестроении (перезапуске дашборда) не терять последние 100 событий, а просто добавлять в начало листа новые и компилировать Markdown.

## 3. Требования к реализации скрипта обновления (`dashboard_engine.py`)
- Запуск: должен уметь работать как демон с `time.sleep(60)`.
- Никакой галлюцинации: Формат файла `STATUS.md` должен 1:1 соответствовать этому документу. Никаких дополнительных "умных" анализов или лишних метрик, только сухие данные.
- State-устойчивость: Запуск/остановка дашборда не должна сбрасывать историю событий. Данные истории должны лежать отдельно в `temp/`.
