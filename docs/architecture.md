# Документ Архитектуры: Экосистема «LightRAG + Antigravity + n8n»

## 1. Роль компонентов системы

*   **Google Antigravity (Центр управления и Vibe-coding):** Основная среда разработки и интерфейс для работы AI-агентов (Claude Code, Gemini 3.1 Pro). Агент в Antigravity обладает субъектностью: он сам пишет код, тестирует его, планирует задачи и обращается к внешним инструментам.
*   **LightRAG (Мозг / Долговременная память):** Инструмент для работы с RAG (Retrieval-Augmented Generation) на базе графов знаний. Выступает в роли «библиотекаря», который хранит историю проектов, решения, архитектурные связи и факты вне контекстного окна агента. Обновляется инкрементально, без пересчета всей базы при добавлении новых файлов.
*   **Neo4j (Графовая база данных):** Хранилище графа знаний (Knowledge Graph). Отвечает за хранение «узлов» (сущностей, фактов, имен, проектов) и «ребер» (связей между ними). Позволяет AI понимать контекст (например, что "разработчик Иван" связан с "проектом Дашборда").
*   **Qdrant (Векторная база данных):** Высокопроизводительное хранилище эмбеддингов (векторных представлений текста). Обеспечивает семантический поиск (поиск по смыслу) для извлечения релевантных кусков текста перед тем, как передать их агенту.
*   **n8n (Оркестратор и автоматизация):** Визуальный low-code конструктор рабочих процессов. Используется для автоматизации рутины, парсинга (например, через Firecrawl), интеграции с Telegram и сторонними API. Выполняет функцию «рук» системы, исполняя детерминированные (предсказуемые) процессы.
*   **MCP Server (Model Context Protocol):** Универсальный протокол-мост, стандартизирующий связь между AI-моделью (Antigravity/Claude) и внешними сервисами (n8n, LightRAG). Позволяет агенту понимать, какие "тулы" (инструменты) ему доступны, и вызывать их автономно без жесткого программирования интеграций.

## 2. Схема взаимодействия (Architecture Overview)

```text
[ Пользователь / Разработчик ]
           │
           ▼
[ Google Antigravity (AI IDE) ] <─── (Локальный контекст, терминал, браузер)
           │
           ├─► (Протокол MCP) ──────┐
           │                        │
           ▼                        ▼
[ MCP Server: LightRAG ]   [ MCP Server: n8n ]
           │                        │
     ┌─────┴─────┐                  │
     ▼           ▼                  ▼
[ Neo4j ]    [ Qdrant ]      [ Рабочие процессы (Workflows) ]
(Связи и     (Векторы и             │
 сущности)    семантика)            ▼
                               [ Внешние API, Telegram, Скраперы ]
```

**Логика работы:**
Агент внутри Antigravity принимает задачу от пользователя. Если ему не хватает контекста о проекте, он через MCP обращается к LightRAG. LightRAG преобразует запрос с помощью LLM (например, Gemma 2.5 Flashlite) и извлекает из Qdrant и Neo4j нужные факты и их связи. Если агенту необходимо выполнить реальное действие (отправить письмо, спарсить сайт), он через MCP отправляет команду в n8n.

## 3. Описание потоков данных (Data Flows)

### Flow 1: Запись в долговременную память (Инкрементальное обучение)
1. Пользователь или n8n-бот (например, аудиосообщение из Telegram, расшифрованное в текст) передает информацию для запоминания.
2. Данные поступают в LightRAG.
3. Встроенная Embedding-модель (например, text-embedding-3) превращает текст в числа (эмбеддинги) и сохраняет в **Qdrant**.
4. LLM (модель-библиотекарь) читает текст, извлекает сущности и связи, формируя узлы и ребра графа, которые сохраняются в **Neo4j**.

### Flow 2: Извлечение контекста (Retrieval)
1. Агент в Antigravity получает запрос пользователя (например: *"Как устроен мой n8n MCP сервер?"*).
2. Antigravity через MCP-соединение вызывает инструмент поиска в LightRAG.
3. LightRAG проходит по графу (Neo4j) и находит связанные кластеры информации.
4. Извлеченный чистый контекст (без мусора) возвращается агенту в Antigravity, не переполняя основное контекстное окно.

### Flow 3: Исполнение задач (Execution)
1. Агент принимает решение обновить данные или спарсить сайт.
2. Antigravity формирует JSON-запрос и через MCP передает команду в n8n.
3. n8n принимает запрос (через Webhook или внутренний триггер), запускает детерминированную автоматизацию.
4. Результат выполнения (например, спарсенные контакты) возвращается обратно агенту или автоматически записывается в базу.

## 4. Правила работы AI агента с архитектурой (Agent Guidelines)

Для успешной работы в экосистеме агент должен соблюдать **«Пирамиду загрузки данных»**:

1. **Сохранение причин, а не только решений:** Самое ценное для графа знаний — почему было принято решение. Если выбран стек Next.js, в мозг записывается факт *«Выбрали Next.js, потому что это дешевле для текущей архитектуры»*.
2. **Запрет на загрузку мусора:** Агенту строго запрещено сохранять в LightRAG логи с ошибками, исходный код целиком (дампы) или неструктурированные базы данных. Мозг предназначен для хранения архитектуры, фактов и стратегий.
3. **Автономная саморефлексия:** При сбое в выполнении задачи (например, ошибка 400/500 от n8n), агент должен пересмотреть свой шаг, проанализировать логи через консоль и самостоятельно исправить отправляемый JSON.
4. **Контекстный инжиниринг (Context Engineering):** Модель должна самостоятельно обращаться к памяти перед началом выполнения сложного блока, чтобы вспомнить актуальные конвенции (например, запрет на определенные компоненты или стили).
5. **Мультиагентность:** Агент должен делегировать рутинные задачи (например, мониторинг отзывов, веб-скрейпинг) в n8n-цепочки, не пытаясь реализовать их на ходу через одноразовые Python-скрипты, чтобы экономить токены и ресурсы.

## 5. Технические детали реализации (Tech Stack & Configs)

*   **Antigravity Config (`mcp_servers.json` / UI):** 
    Конфигурационный файл или раздел интерфейса, отвечающий за подключение MCP-серверов. 
    Актуальная структура для подключения LightRAG (запуск внутри Docker):
    ```json
    {
      "mcpServers": {
        "lightrag": {
          "command": "docker",
          "args": ["exec", "-i", "lightrag_api", "python", "/scripts/mcp_lightrag.py"]
        },
        "n8n": {
          "command": "npx",
          "args": ["-y", "@n8n/mcp-server"]
        }
      }
    }
    ```
    Это обеспечивает изоляцию процессов от хост-системы Mac.

*   **Инфраструктура баз данных (Self-hosted через Docker):**
    *   **Neo4j:** Рекомендуемый порт по умолчанию `7474` (HTTP/UI) и `7687` (Bolt протокол для запросов графа).
    *   **Qdrant:** Доступ по порту `6333` (REST API) и `6334` (gRPC). Хранит векторы, размерность которых зависит от выбранной embedding-модели (например, 1536 для OpenAI text-embedding-3-small).
    *   **n8n:** Стандартный порт `5678`. Настроен на работу через реверс-прокси (с поддоменами вида `n8n.yourdomain.com`) для получения Webhook-ов.

*   **API Эндпоинты (Webhooks в n8n):**
    n8n ожидает входящие запросы через Webhook узлы.
    Формат URL: `https://[n8n-домен]/webhook/[название-пути]`.
    Аутентификация: **Bearer Token** или **Header Auth**, настраиваемая внутри n8n узла Webhook. Метод вызова — `POST` (передача JSON в Body).

*   **Модели под капотом LightRAG:**
    *   *LLM:* qwen2.5-coder:7b (через Ollama local).
    *   *Embeddings:* locally computed or via provider as configured in .env.
