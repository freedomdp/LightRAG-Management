# Экспертная Система: HLV + Spec-Driven Development (Zero-Compression Reference)

Данный документ является сводным техническим эталоном для реализации архитектуры **Human-LLM-Validation (HLV)** и методологии **Spec-Driven Development (SDD)** в проекте LightRAG-Management.

---

## 🏛️ 1. Архитектурные Слои (The 6-Layer Cycle)

Процесс разработки больше не является линейным написанием кода. Это циклический процесс валидации спецификаций:

1.  **Constitution (Устав)**: Мастер-стандарт проекта (стек, стили, ограничения).
2.  **Specification (Спекуляция)**: Описание функциональности. *Источник правды №1.*
3.  **Clarify (Сессия вопросов)**: Фиксация неопределенностей (GAP Registry).
4.  **Intent/Plan (Технический проект)**: Архитектурное решение задачи.
5.  **Task Parsing (Дерево задач)**: Трансформация плана в атомарные TODO с зависимостями.
6.  **Implementation (Реализация)**: Написание кода под надзором HLV-гейтов.

---

## 📜 2. Контрактная Генерация (VibeContract)

Перед каждым блоком кода Агент обязан зафиксировать **VibeContract**:
- **Pre-conditions**: Что приходит на вход (строгая типизация).
- **Post-conditions**: Что выходит (гарантия формата).
- **Invariants**: Состояние системы, которое нельзя нарушать.
- **Exceptions**: Все сценарии ошибок, которые *должны* быть обработаны.

---

## 🛡️ 3. Защита от Дрейфа (Drift Guard)

Для борьбы с волатильностью ИИ применяются следующие механизмы:
- **Spec-Anchored**: Спецификация обновляется ПЕРЕД кодом. Ручная правка кода без обновления спеки — запрещена.
- **Compiler-Driven Schemas**: Схемы инструментов для ИИ (Tool definitions) генерируются автоматически из кода (Agentica style).
- **Continuous Validation**: Гейты HLV работают на постоянной основе.
- **Surgical Ingest**: Под каждую задачу подгружается только минимальный набор артефактов (D, R, S, M, Data, ADR, T).

---

## ⚔️ 4. Жесткие гейты (HLV Logic)

- **Gate Passed**: Задача завершена только при получении бинарного ответа от внешнего валидатора (линтер, тест, бинарник hlv).
- **Gate Failed**: Агент блокируется. Разрешено только исправление ошибки через пересмотр Плана или Спецификации.
- **Wipe Strategy**: При зацикливании (3+ неудачных попытки прохода гейта) — полная очистка контекста и возвращение на этап `Clarify`.

---

## 🔍 5. Работа с Legacy (Baseline Concept)

Для существующих проектов работа начинается с создания **Baseline**:
1.  Снятие слепка API и интерфейсов.
2.  Фиксация текущей архитектуры.
3.  Любые изменения оцениваются на "соответствие Baseline" (Alignment check).
