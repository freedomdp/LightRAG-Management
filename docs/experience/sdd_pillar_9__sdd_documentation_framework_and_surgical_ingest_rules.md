# SDD Pillar 9: SDD-Documentation Framework and Surgical Ingest Rules

### 9. SDD-Documentation: Слой артефактов (Habr)

**Концепция**: Внедрение "Жесткой Инженерии" через вынос всех знаний во внешнюю, строго структурированную документацию, которая заменяет длинные и шумные промпты.

**7 Классов Артефактов (The 7 Pillars of SDD-Doc)**:
1.  **D (Description)**: Общее описание фичи (ТЗ).
2.  **R (Requirements)**: Список функциональных и нефункциональных требований.
3.  **S (Specification)**: Архитектурные детали, API, структуры данных.
4.  **M (Milestones)**: План реализации (Major Tasks).
5.  **Data (Data Schema)**: Описание БД и потоков данных.
6.  **ADR (Architectural Decision Records)**: Почему было принято решение X, а не Y.
7.  **T (Tests/Validation)**: Описание гейтов и условий прохождения "HLV-вальса".

**Правила "Хирургического Инджеста" (Surgical Ingest)**:
- Агент никогда не грузит "всё" сразу. Под каждую задачу подбирается только необходимый набор артефактов.
- **GAP Registry**: Если в артефактах обнаружена дыра (GAP), агент обязан создать запись в реестре неопределенности и попросить уточнения, а не додумывать самому.

**Механика "Нативного исполнения"**:
- Тяжелые проверки (линтеры, билды, безопасность) выносятся из LLM в нативные скрипты (hlv-бинарники). 
- ИИ получает только бинарный ответ: "Gate Passed" или "Gate Failed + Log".

**Связь с HLV**: Эта система артефактов и есть "Спецификация", против которой hlv проверяет код.
