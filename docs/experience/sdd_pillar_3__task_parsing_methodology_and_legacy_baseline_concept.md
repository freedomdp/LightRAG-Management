# SDD Pillar 3: Task Parsing Methodology and Legacy Baseline Concept

### 3. Task Parsing: Методология превращения Планов в Задачи (Habr)

**Концепция**: План (Human-readable) должен быть трансформирован в структурированный список задач (Machine-parsable Tasks), чтобы агент мог выполнять их последовательно, сохраняя контекст файла.

**Алгоритм парсинга (The Task Parsing Logic)**:
1.  **Дерево зависимостей**: Определение порядка файлов. Сначала базовые классы/интерфейсы, затем бизнес-логика, затем тесты.
2.  **Surgical Changes (Хирургические правки)**: Каждая задача должна описывать точечное изменение (например, "добавить метод X в класс Y"), а не "переписать файл".
3.  **Traceability (Трассируемость)**: Каждая задача должна содержать ссылку на пункт в Спецификации, который она реализует.

**Концепция Baseline (Базовой линии)**:
- При работе с существующим (legacy) проектом, агент сначала создает **Baseline-файл** — слепок текущего состояния (README, дерево файлов, ключевые интерфейсы).
- Любая правка оценивается относительно Baseline.
- Если код в проекте противоречит Baseline — это повод для "Wipe" (сброса) и пересмотра ТЗ, а не попытка фикса "на лету".

**Связь с HLV**: Каждая атомарная задача завершается гейтом ("Gate Passed"). Если гейт не пройден, агент блокируется до исправления, не переходя к следующей задаче.
