# SDD Pillar 1: GitHub Spec-Driven Development Life Cycle

### 1. GitHub SDD Framework (Spec-Driven Development)

**Концепция**: Переход от "Код как источник правды" к "Спецификация как источник правды". Код — это производный артефакт, генерируемый из спецификации.

**Детальный цикл разработки (6 слоев)**:
1.  **Constitution (Конституция)**: Глобальные правила проекта, стек, архитектурные ограничения. Основа для всех последующих решений.
2.  **Specification (Спецификация)**: Описание *что* должно быть сделано на языке бизнеса и архитектуры. Не содержит кода.
3.  **Clarify (Уточнение)**: Этап вопросов и ответов между ИИ и человеком для устранения двусмысленностей в спеке.
4.  **Intent/Plan (План реализации)**: Описание *как* это будет сделано технически (выбор алгоритмов, структур данных).
5.  **Task Parsing (Атомарные задачи)**: Разбиение плана на список дел (TODO), где каждый пункт — это изменение в конкретном файле.
6.  **Implementation (Реализация)**: Написание кода строго по списку задач.

**Правило "Чек-поинтов" (Checkpoints)**:
- Агент ОБЯЗАН останавливаться после выполнения каждой логической задачи или этапа (например, после создания Плана).
- Кодинг не начинается, пока План не утвержден человеком.
- После каждого блока изменений проводится валидация (Layer 3 HLV).

**Связь с HLV**: Каждая задача в списке должна иметь ассоциированный с ней "гейт" (тест или маркер), который проверяется внешним валидатором.
