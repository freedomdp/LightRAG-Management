# Сравнение моделей: Qwen 2.5 Coder vs Gemma 4 E4B-it

Этот анализ помогает выбрать оптимальную модель для разных этапов жизни нашего проекта.

| Критерий | Qwen 2.5 Coder (7B) | Gemma 4 E4B-it (4B) |
| :--- | :--- | :--- |
| **Параметры** | 7 миллиардов параметров | 4 миллиарда (эффективных) |
| **Сильные стороны** | **Код и структура**. Понимает JSON, ER-экстракцию на 5+. | **Рассуждения и Мультимодальность**. Сильнее в планировании и диалоге. |
| **Контекст** | 32K (стабильно). | **128K (нативно)**. Позволяет "видеть" весь проект целиком. |
| **Function Calling** | Хорошо (через промпты). | **Нативно (встроено)**. Идеально для управления инструментами MCP. |
| **Потребление ОЗУ** | Около 4.7 ГБ. | Около 9.6 ГБ (из-за сложной архитектуры весов). |

## Резюме для нашего проекта

### 1. Почему оставляем Qwen для индексации
Индексация LightRAG требует **консистентности**. Если мы сменим модель на 53%, мы получим "винегрет" из сущностей. Qwen уже "привык" к нашему стилю данных — давайте дойдем до 100% на нем.

### 2. Почему переключаем Агента на Gemma 4
Для роли "Менеджера-Архитектора" (Manager-Architect) нам не нужен кодер, нам нужен **стратег**. Gemma 4 лучше справляется с поиском логических ошибок в ТЗ и планированием этапов миграции на Rydzen 9.

### 3. Эффект таймаутов
Мы видели в логах `httpx.ReadTimeout` у Qwen (7B). Gemma 4 (4B) легче на подъем и должна отвечать быстрее при больших объемах входных данных.
