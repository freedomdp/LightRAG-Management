## 🏗️ ARCHITECTURAL CYCLE (SDD + HLV)
Все изменения в системе n8n проходят через обязательный цикл архитектора. Это гарантирует отсутствие ошибок и полное соответствие стандартам проекта.

1. **Constitution (project.md)**: Сверка с целью проекта. Не противоречит ли новый узел общей архитектуре?
2. **[SPECIFICATION]**: Четкое описание — ЧТО делаем и ПОЧЕМУ. Описание логики работы (например: "Нужна пагинация, потому что записей > 100").
3. **[PLANNING]**: Техническая декомпозиция. Список необходимых узлов n8n, названия ключей API, структура JSON.
4. **[VALIDATION] ← Human Approval**: **КРИТИЧЕСКАЯ ОСТАНОВКА**. Предъявление плана пользователю. Без явного "OK" переход к этапу Execution запрещен.
5. **[EXECUTION]**: Создание атомарных задач в `task.md`. Сборка воркаута по шагам.
6. **[VERIFICATION]**: 
    - Тесты через `Execute Workflow`.
    - Сверка со стандартами `project-standards`.
    - Проверка по `Architecture Review Checklist` (из конца этого документа).
7. **[LEARNING]**: Фиксация опыта через `log_new_experience` и Git Commit.

## 🛠️ ГЛУБОКИЙ РЕФЕРЕНС: 25 КЛЮЧЕВЫХ УЗЛОВ

### 1. AI Agent (The Brain)
Основной узел для запуска ЛЛМ с инструментами. Это "сердце" любой интеллектуальной автоматизации.

#### 🛠️ Основные параметры:
- **Agent Type**: 
  - `Conversational Agent`: Использует Buffer Memory для диалога. Помнит контекст беседы.
  - `Tools Agent`: Анализирует задачу и выбирает подходящий инструмент из подключенных.
  - `Plan & Execute`: Новинка v2. Разбивает сложную задачу на подзадачи, выполняет их и собирает результат.
- **Model**: Подключение провайдеров (Ollama, OpenAI, Anthropic).
- **Options (Параметры модели)**: 
  - `Temperature`: (От 0 до 2). 0 — для строгой логики, 0.7+ — для креативности.
  - `Max Tokens`: Лимит длины ответа (предотвращает бесконечные траты токенов).
  - `Top P`: Контроль разнообразия слов.
  - `Presence Penalty`: Штраф за повторение слов.
  - `Frequency Penalty`: Штраф за частое использование одних и тех же фраз.

#### 📝 Prompts: 
- **System Message**: Критически важная роль (Persona). Например: "Ты профи-копирайтер, пиши в стиле Х".
- **Context**: Инъекция данных из предыдущих узлов (RAG результаты, данные CRM).

#### ⚙️ Продвинутые опции (Advanced): 
- **Max Iterations**: Лимит попыток агента решить задачу (по умолчанию 5). Рекомендуется 3-10.
- **Return Intermediate Steps**: Если включено, возвращает весь ход мыслей агента (Chain of Thought).
- **Callback URL**: Для внешнего мониторинга работы агента.

### 2. HTTP Request (The Connector)
Узел для взаимодействия с любыми внешними API. Позволяет интегрировать n8n с чем угодно.

#### 🛠️ Основные параметры:
- **Method**: 
  - `GET`: Получение данных.
  - `POST`: Создание ресурсов/передача данных.
  - `PUT / PATCH`: Обновление существующих данных.
  - `DELETE`: Удаление.
- **URL**: Полный адрес запроса (поддерживает Expression, например: `https://api.site.com/{{ $json.id }}`).
- **Authentication**: 
  - `Predefined Credential`: Использование сохраненных в n8n ключей.
  - `Generic Auth`: Basic, Header, Query Param, OAuth2.
- **Send Body**: 
  - `JSON`: Стандарт для API.
  - `Form-Data`: Для загрузки файлов.
  - `Raw`: Произвольный текст/XML.

#### ⚙️ Полный список опций (Options): 
- **Batching**: 
  - `Batch Size`: По сколько запросов слать за раз.
  - `Batch Interval`: Пауза между пакетами (мсек).
- **Retry on Fail**: 
  - `Max Retries`: Количество попыток (до 5).
  - `Wait Between Retries`: Задержка перед повтором.
- **Ignore SSL Issues**: Для работы с самоподписанными сертификатами.
- **Response Format**: `JSON`, `String`, `Binary`, `File`.
- **Query Parameters**: Удобный список для `?key=value`.
- **Header Parameters**: Список заголовков (например, `Content-Type`, `Custom-Agent`).
- **Proxy**: Настройка прокси-сервера для скрытия IP.
- **Redirects**: `Follow` / `Manual` / `Error`.

### 3. Webhook (The Entry Point)
Входная точка для внешних сервисов.

#### 🛠️ Основные параметры:
- **HTTP Method**: `GET`, `POST`, `PUT`, `PATCH`, `DELETE`.
- **Path**: Уникальное имя вебхука (автоматически формирует URL).
- **Authentication**: `None`, `Basic Auth`, `Header Auth`.
- **Response Mode**: 
  - `Immediately`: Отвечает 200 OK сразу при получении.
  - `When Last Node Finishes`: Ответ содержит JSON последнего узла в флоу.
  - `On Click`: Ответ через специальный узел Respond to Webhook.

#### ⚙️ Опции (Options): 
- **Binary Data**: Позволяет принимать файлы. Поле `Property Name` определяет, где будет лежать файл.
- **Ignore Bots**: Игнорировать запросы от поисковых роботов.
- **No Response**: Не посылать никакого ответа.
- **Response Content Type**: Например, `application/xml` или `text/html`.

### 4. Code (The Powerhouse)
Узел для написания произвольного JS (Node.js).

#### 🛠️ Основные параметры:
- **Language**: `JavaScript` (стандарт).
- **Mode**: 
  - `Run Once for Each Item`: Код выполняется в цикле для каждого объекта. Удобно для простых преобразований.
  - `Run Once for All Items`: Получает весь массив `items`. Позволяет фильтровать, аггрегировать и пересобирать данные.

#### 📝 API & Variables: 
- `$json`: Доступ к полям.
- `$itemIndex`: Порядковый номер.
- `$workflow`: Метаданные воркаута.
- `$executionId`: Уникальный ID запуска.
- `helpers.getBinaryDataBuffer(item, prop)`: Работа с бинарными буферами.

### 5. Loop Over Items (The Engine)
Разбиение массива на части для пошаговой обработки.

#### 🛠️ Основные параметры:
- **Batch Size**: Количество элементов в одном проходе.
- **Reset**: Сброс счетчика при каждом новом входе в цикл (если есть вложенные циклы).

### 6. IF (The Decision Maker)
Логическое разветвление потока.

#### 🛠️ Основные параметры:
- **Conditions**: 
  - `String`: Equal, Not Equal, Contains, Starts With, Regex.
  - `Number`: Smaller, Greater, Equal, Is NaN.
  - `Boolean`: Is True/False.
  - `Data`: Позволяет сравнивать даты напрямую.
- **Combine Conditions**: Выбор логики `AND` (все условия верны) или `OR` (хотя бы одно).

### 7. Switch (The Router)
Многопоточный распределитель.

#### 🛠️ Основные параметры:
- **Mode**: 
  - `Expression`: Проверка значения одного поля.
  - `Rules`: Набор сложных условий для разных выходов.
- **Fallback Output**: Куда пойдут данные, если ни одно правило не сработало.

### 8. Merge (The Data Joiner)
Узел для объединения данных из двух разных веток.

#### 🛠️ Основные параметры:
- **Mode**: 
  - `Append`: Простое сложение массивов (Вход 1 + Вход 2).
  - `Combine`: Соединение записей.
    - *Merge by Position*: 1-я с 1-й, 2-я со 2-й.
    - *Merge by Key*: Соединение по общему полю (например, `id`). Это полноценный SQL JOIN.
  - `Choose`: Пропуск только одной ветки (Main или Branch).
  - `Remove Key Matches`: Удаление из Входа 1 тех записей, что есть во Входе 2.

#### ⚙️ Опции (Options): 
- **Multiple Matches**: Что делать, если ключу соответствует несколько записей? (`Match all`, `Match first`).
- **Always Output Data**: Если одна ветка пустая, воркаут не остановится, а выдаст пустой результат.

## 📐 16 КЛАССИЧЕСКИХ АЛГОРИТМОВ АВТОМАТИЗАЦИИ (Deep Dive)

### 1. Anti-Looping (Защита от бесконечных циклов)
Алгоритм для предотвращения повторной обработки одних и тех же данных.
1. **Trigger**: Получение данных (например, сообщение из ТГ).
2. **Check**: Узел `Data Table` или `Google Sheets` ищет `ID` сообщения.
3. **Condition**: Если `ID` найден — воркаут завершается (`Stop`).
4. **Execution**: Если `ID` не найден — выполняются действия.
5. **Final**: `ID` записывается в таблицу как "обработанный".

### 2. Exponential Backoff (Интеллектуальные повторы)
Позволяет системе «остыть» перед повторным запросом к API.
1. **Error Handler**: Узел `Error Trigger` ловит сбой.
2. **Calculation**: `Code Node` вычисляет задержку: `delay = Math.pow(2, attempts) * 5000`.
3. **Pause**: Узел `Wait` использует результат из `delay`.
4. **Retry**: `Execute Workflow` запускает задачу заново.

### 3. Split-Parallel-Aggregate (Масштабируемость)
Обработка 1000 элементов максимально быстро.
1. **Split**: `Loop Over Items` с Batch Size = 50.
2. **Execute**: Вызов под-воркаута в режиме `Asynchronous`.
3. **Wait**: Узел `Wait` (Webhook mode) ждет сигнала о готовности всех частей.
4. **Merge**: `Aggregate` собирает все ответы в итоговый массив.

### 4. Dirty Data Cleaning & Normalization (Очистка)
Приведение любого входящего мусора к стандарту.
1. **Catch**: Получение сырых данных (Webhook).
2. **First Pass**: `Code Node` удаляет HTML-теги, лишние пробелы и спецсимволы.
3. **JSON Schema**: Проверка соответствия структуры через `IF/Switch`.
4. **Mapping**: Приведение полей к единому виду (например, все телефоны к формату `+7...`).

### 5. API Cursor-Based Pagination (Пагинация)
Стандарт для выгрузки больших объемов данных из CRM/ERP.
1. **Init**: Переменная `cursor = null`.
2. **Request**: HTTP запрос с параметром `cursor`.
3. **Process**: Обработка полученных 100 записей.
4. **Check**: Если в ответе есть `next_cursor` — `cursor = next_cursor` и переход в начало цикла `Loop`.
5. **End**: Если `next_cursor` пуст — выход из цикла.

### 6. Notification Cascade (Отказоустойчивость уведомлений)
Гарантия доставки критической информации.
1. **Primary**: Отправка сообщения в Telegram.
2. **Delay**: Узел `Wait` на 10 минут.
3. **Check**: Запрос к API Telegram / базе данных: "Прочитано ли сообщение?".
4. **Fallback**: Если не прочитано — отправка в Email / SMS.
5. **Final**: Логгирование инцидента.

### 7. Dual Validation (Guardrails / Качество ИИ)
Защита от галлюцинаций и ошибок модели.
1. **Generator**: ИИ-агент создает текст/код.
2. **Validator**: Второй узел ИИ получает промпт: "Проверь этот текст на ошибки и факты. Ответь только JSON {valid: boolean, errors: string}".
3. **Logic**: Узел `IF`. Если `valid == true` — идем дальше. 
4. **Retry**: Если `false` — возвращаем текст на исправление первому агенту с указанием ошибок.

### 8. State Persistence (Глобальная память)
Таймштампы и флаги между разными запусками.
1. **Get**: Узел `Code` читает `Static Data`: `lastRun = staticData.lastDate`.
2. **Filter**: Запрос к источнику с фильтром `date > lastRun`.
3. **Execute**: Выполнение действий над новыми данными.
4. **Update**: Узел `Code` перезаписывает `staticData.lastDate = now`.

### 9. Binary to Text AI Pipeline (Обработка документов)
Автоматическое чтение входящих файлов.
1. **Receive**: Webhook принимает PDF/Docx.
2. **Convert**: Узел `Extract from File` превращает бинарные данные в текст.
3. **Clean**: `Code Node` удаляет мусорные символы и колонтитулы.
4. **Digest**: ИИ-агент делает краткое содержание (Summary).

### 10. Intelligent Rate Limiting (Обход блокировок)
Безопасная работа с жесткими лимитами (например, 1 запрос в минуту).
1. **Queue**: Сбор всех задач в массив.
2. **Loop**: Начало цикла `Loop Over Items` с Batch Size = 1.
3. **Action**: Выполнение запроса к API.
4. **Wait**: Узел `Wait` на фиксированное время (превышающее лимит API).
5. **Loop Return**: Возвращение за следующим элементом.

### 11. Dynamic Routing (Умная диспетчеризация ИИ)
Экономия ресурсов за счет выбора нужной модели.
1. **Classifier**: Маленькая быстрая модель (например, 7B) анализирует запрос.
2. **Switch**: Если запрос простой — идем в `Groq/Ollama`. Если сложный — переключаемся на `GPT-4 / Claude 3`.
3. **Logic**: Настройка порогов сложности (Complexity Score).

### 12. Lead De-duplication & Merging (Чистка CRM)
1. **Search**: Поиск нового Email в Google Sheets / CRM.
2. **IF**: Если найден — обновляем существующую запись (`Update Row`).
3. **Else**: Если не найден — создаем новую (`Append Row`).
4. **Merge**: Слияние интересов пользователя (например, прикрепить новый лид к существующей компании).

### 13. Content Summarization Cascade (Long-reads)
Обработка текстов на 20к+ слов.
1. **Split**: Узел `Code` режет текст на куски по 3000 символов.
2. **Loop**: Каждый кусок обрабатывается ЛЛМ (`AI Agent`).
3. **Aggregate**: Собираем краткие выжимки кусков в один массив.
4. **Final Summarize**: ИИ делает «сводку сводок» для итогового отчета.

### 14. Professional Health Check / Pinging
Мониторинг доступности API.
1. **Schedule**: Запуск каждые 5 минут.
2. **Request**: HTTP GET запрос к вашему API.
3. **Options**: `Full Response = true`.
4. **IF**: Если `status != 200` — запуск **Notification Cascade** (см. алгоритм №6).

### 15. Multi-Step Human-in-the-Loop Approval
Остановка воркаута для ручной проверки.
1. **Process**: Генерация чего-то важного (например, счета).
2. **Notify**: Отправка сообщения в ТГ с кнопками "Одобрить" / "Отклонить".
3. **Wait**: Узел `Wait` (Webhook mode) замирает.
4. **Resume**: Когда администратор нажимает кнопку, воркаут возобновляется с результатами клика.

### 16. Workflow Integrity & Connectivity (Целостность потока)
Решение проблемы «оторванных» или не сработавших узлов.
1. **Принцип Явной Связи**: Если узел должен передать данные, он обязан иметь входящую линию. Если узел выполняет «фоновое» действие, используйте узел `Wait` (1 сек) или `Merge`, чтобы соединить его с основной магистралью.
2. **Cross-Node Referencing**: Чтобы данные из начала флоу были доступны в конце (даже если прямого пути нет), используйте синтаксис: `{{ $('Trigger Node Name').item.json.field }}`. Это позволяет «пробрасывать» данные сквозь любые разрывы.
3. **Explicit Merge Gate**: Если у вас есть параллельные ветки (например, одна пишет в лог, другая в ТГ), используйте узел `Merge` в режиме `Wait for both to finish`. Это гарантирует, что следующий узел не запустится, пока ОБЕ «оторванные» ветки не завершатся.
4. **Final Node Requirement**: Каждый воркаут должен иметь «Конечную точку истины»: узел `Respond to Webhook`, финальное сообщение в Telegram или запись в БД с флагом `status: finished`. Если ваша цепочка обрывается «в пустоту» — это потенциальный баг.
5. **Debug Validation**: В истории запусков (Execution History) ищите узлы серого цвета — это те, до которых «не дошел сигнал». Если видите такой узел, значит, логика `IF/Switch` перед ним увела поток в другую сторону или связь физически отсутствует.

## 🛡️ ARCHITECTURE REVIEW CHECKLIST (АНТИ-ГАЛЛЮЦИНАЦИЯ)
Используйте этот чек-лист для проверки любого воркаута (особенно сгенерированного ИИ) перед запуском.

### 1. Проверка связности (Connectivity)
- [ ] **Физическая связь**: Каждый узел (кроме триггеров) имеет входящую линию?
- [ ] **Логический финал**: Есть ли в конце каждой ветки узел-результат (`Respond to Webhook`, `Telegram Send`, `Sheet Append`)? Воркаут не должен обрываться "в никуда".

### 2. Валидация Выражений (Expressions)
- [ ] **Существование узлов**: Все обращения вида `{{ $('Specific_Node')... }}` ссылаются на РЕАЛЬНО существующие и правильно названные узлы? (ИИ часто выдумывает имена типа "DataMapper").
- [ ] **Метод обращения**: Используется ли правильный синтаксис для доступа к данным?
    - `{{ $json.field }}` — для текущего входа.
    - `{{ $('Node').item.json.field }}` — для обращения к конкретному узлу.

### 3. Схема данных (Data Schema)
- [ ] **Сопоставление полей**: Ожидает ли узел Б те данные, которые выдает узел А? (Например, если узел Б ждет `ID` как число, а узел А выдает его как строку в кавычках).
- [ ] **Массивы vs Объекты**: Если на вход пришел массив, а узел умеет работать только с одним объектом, стоит ли перед ним `Split in Batches` или `Aggregate`?

### 4. Бинарные данные (Files)
- [ ] **Property Name**: Если передается файл, во всех узлах указано одинаковое имя бинарного поля (например, `data`)?
- [ ] **Read/Write**: Есть ли узел `Extract from File`, если нужно прочитать текст из PDF, или это просто пересылка файла "как есть"?

### 5. Безопасность и Лимиты
- [ ] **API Limits**: Есть ли узлы `Wait`, если воркаут делает более 1-2 запросов за раз к чувствительным API?
- [ ] **Credentials**: Указаны ли названия учетных записей или оставлены заглушки `[ENTER_YOUR_API_KEY]`?
- [ ] **Error Handling**: Что произойдет, если узел [X] упадет? Есть ли у него `Retry` или общий `Error Workflow`?

### 6. Анти-Цикл (Loop Safety)
- [ ] **Return Path**: Возвращается ли связь из конца цикла обратно в узел `Loop Over Items`?
- [ ] **Termination**: Есть ли выход из цикла по условию `Done`?
