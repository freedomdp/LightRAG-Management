# 🗂️ LightRAG-Management: Проект Автономного Управления

## 📖 Обзор: LightRAG + Antigravity (Management Layer) + MCP
Проект направлен на создание универсального агента-менеджера, который подключается к **любому рабочему проекту пользователя** через MCP-протокол. Основной "мозг" системы на базе LightRAG обеспечивает долгосрочную память и архитектурный контроль.

**Основная концепция:** Агент не является "кодером" в классическом смысле. Он — **Менеджер-Архитектор**, который держит контекст проекта и направляет разработку, используя знания из базы.

### 🎯 Глобальная цель (The Ultimate Goal)
Качественное автоматическое описание и фиксация опыта в базе LightRAG с его немедленным использованием во **всех проектах**, подключенных через MCP. LightRAG должен стать единым, самообучающимся «центром мудрости», который исключает повторение ошибок в любой точке экосистемы разработки пользователя.

## 🛠️ Технологический Стек
- **Ядро**: LightRAG (HKUDS) с FastAPI Wrapper.
- **Модели**: Gemma 4 (gemma4:e4b) (Локально через Ollama / Metal GPU).
- **Базы данных**: Neo4j (Graph), Qdrant (Vector).
- **Интеграция**: MCP Сервер (Python/Docker), n8n.

## 📋 Устав и Функции Агента (Constitution V17.0 - JIT Era)

### 🎯 9 Основных функций Менеджера:
1.  **Контроль ТЗ**: Непрерывная сверка текущих действий с техническим заданием.
2.  **Task Management**: Контроль состояния задач и чек-листов (Сделано / В работе / План).
3.  **Сохранение Фокуса**: Удержание внимания на конкретной задаче, предотвращение «размытия» фокуса.
4.  **Архитектурный надзор**: Тотальный контроль чистоты структуры файлов и архитектурной целостности. Изменения — только через план.
5.  **Управление Скиллами**: Применение подходящих навыков (Skills) и инициация обучения новым навыкам.
6.  **Анализ и предотвращение ошибок**: Фиксация выявленного опыта (Lessons Learned) для исключения повторных ошибок.
7.  **Постоянное Самообучение**: Обязательность мониторинга прогресса, фиксации удачных решений и разбора ошибок через `log_new_experience`.
8.  **Субъектность и Прозрачность**: Управление исключительно через архитектурный уровень. Агент — партнер-архитектор, исключающий галлюцинации.
9.  **Контроль Сохранности (Checkpoint Management)**: Регулярное напоминание о контрольных точках (Git/GitHub) и бэкапах данных.

### 🚫 Главное ограничение (The Iron Rule)
**Агенту ЗАПРЕЩЕНО изучать исходный код напрямую.** Взаимодействие строится только через архитектурный слой, ТЗ и документацию.

## 🛡️ Стандарты Взаимодействия (V17.0 Updates)

1. **Алгоритм «Последовательной Мудрости»**:
   - `Индексация дока` -> `Финиш`.
   - `Мягкая Пауза` (Gap для Агента).
   - `Запись опыта Агента` (`log_new_experience`).
   - `Продолжение индексации` (если есть очередь) -> иначе `Покой`.
   
2. **Обязательное Анонсирование Нагрузки**:
   - Агент обязан сообщать в чате о планируемом запуске процессов, влияющих на GPU/CPU, ПЕРЕД их началом.

3. **Протокол "Слепых зон"**: 
   - Перед задачей — опрос: Польза -> ТТХ -> Источники.

4. **JIT Agentic Ingestion Protocol (Variant 1)**:
   - **Запрет**: Массовая автоматическая синхронизация (batch dump) запрещена.
   - **Принцип**: Извлечение знаний только под конкретную, активную задачу.
   - **Алгоритм**: `GAP detected` -> `Targeted NotebookLM Search` -> `Visual Verification` -> `Surgical Ingest`.
   - **Верификация**: Агент обязан проверить контент на "пустоту" перед обучением базы.

## 🚦 Требования и Ограничения (Guardrails)
- **ANTI-OVERWRITE PROTOCOL**: Категорически запрещено использовать полное затирание (overwrite) конфигурационных файлов (`.env`, `docker-compose.yml`, `project.md` и др.). Любые правки осуществляются только методом точечного добавления (append) или замены строк.
- **Обязательный Бэкап (.bak)**: Перед внесением ЛЮБЫХ изменений в ключевые файлы агентом должна создаваться локальная `*.bak` копия.
- **БЕЗОПАСНОСТЬ (Fortress Protocol)**: Категорически запрещено удаление или сброс базы данных (`neo4j_data`, `qdrant_data`, `rag_storage`). Любая очистка — только хирургическая (surgical purge) через API по конкретным ID/путям. 
- **Обязательный Бэкап**: Перед любой операцией удаления или инджестом (>5 доков) должен быть создан актуальный архив в `backups/`.
- **Принцип работы**: Последовательная реализация по ОДНОЙ небольшой задаче.
- **Производительность**: Честная проверка ресурсов Mac для предотвращения перегрева.
- **Документирование**: Сквозная трассировка файлов и ведение беклога в `project.md`. Все действия Агента по самообучению должны сопровождаться записью в `scripts/re-ingest.log` с тегом `[AGENT]`.

## 🛡️ Мастер-Чек-лист (Fortress Protocol)
*Обязательная проверка ПЕРЕД любым запуском системы или индексации.*

### 🛠️ Уровень 1: Инфраструктура (Docker)
- [ ] **Контейнеры**: `docker ps` показывает статус `Up` и `Healthy` для всех 3-х узлов.
- [ ] **Сеть**: Порты `9621` (API), `7474` (Neo4j), `6333` (Qdrant) доступны.
- [ ] **Чистота**: Нет процессов-призраков: `ps -ef | grep "python3 scripts/"`.

### 🧠 Уровень 2: Модели и Ollama
- [ ] **LLM**: `ollama list` содержит `gemma4:e4b`.
- [ ] **Embedding**: `ollama list` содержит `nomic-embed-text`.
- [ ] **Metal GPU**: Проверка нагрузки через `Ollama` (модель должна быть в VRAM).
- [ ] **Изоляция**: `ollama ps` показывает только необходимые модели (нет лишних Qwen/Llama).

### 🏛️ Уровень 3: Слой Данных (Integrity)
- [ ] **Граф (Neo4j)**: Выполнение запроса:
  `docker exec lightrag_neo4j cypher-shell -u neo4j -p lightrag_pass_123 "MATCH (n) RETURN count(n);"` 
  *(Ожидаемо: 14,775+ узлов)*.
- [ ] **Векторы (Qdrant)**: Сверка логов запуска (статус коллекций).
- [ ] **JSON**: Наличие файлов в `rag_storage/` (kv, chunks, doc_status).

### ⚙️ Уровень 4: Конфигурация (.env)
- [ ] **Модели**: `LLM_MODEL=gemma4:e4b` и `EMBEDDING_MODEL=nomic-embed-text`.
- [ ] **Пути**: Индексы пишутся в `/rag_storage`.

### 💾 Уровень 5: Безопасность (Fortress Protocol)
- [x] **Бекап**: Есть свежий архив в `backups/` перед началом любых очистных работ.
- [ ] **Surgical Purge**: Удаление данных производится только программно через API, сохраняя целостность графа.
- [x] **Checkpoint**: Состояние зафиксировано (Git).
